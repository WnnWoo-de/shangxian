# 皖盛布碎资源结算系统 - 数据库说明文档

## 项目概述
本项目是一个基于 Vue 3 和 Pinia 状态管理的布碎资源结算系统，使用 LocalStorage 作为数据持久化方案。系统包含客户管理、品类管理、布料管理、单据管理等核心功能。

## 数据存储架构
系统采用 LocalStorage 作为主要存储方案，通过 Pinia 状态管理库管理应用状态。所有数据模型均通过统一的 storage 工具类进行序列化和存储。

## 数据存储键名
| 存储键名 | 数据类型 | 说明 |
|---------|---------|------|
| `wsbs_token` | 字符串 | 用户登录 token |
| `wsbs_user` | 对象 | 用户信息 |
| `wsbs_customers` | 数组 | 客户列表 |
| `wsbs_categories` | 数组 | 品类列表 |
| `wsbs_fabrics` | 数组 | 布料列表 |
| `wsbs_bills` | 数组 | 单据列表 |
| `wsbs_draft_bill` | 对象 | 草稿单据 |
| `wsbs_recent_customers` | 数组 | 最近使用客户 |
| `wsbs_recent_fabrics` | 数组 | 最近使用布料 |
| `wsbs_app_config` | 对象 | 应用配置 |
| `wsbs_stats_cache` | 对象 | 统计缓存 |

#### 存储位置
`wsbs_bills` (LocalStorage)

#### 字段说明
```javascript
{
  id: 'bill-1234567890',            // 单据ID
  type: 'purchase',                 // 单据类型 (purchase/sale)
  billNo: 'B1234567890',            // 单据编号

  createdAt: '2024-01-01 10:30:00', // 创建时间
  updatedAt: '2024-01-01 10:30:00', // 更新时间
  partnerId: 'cust-001',            // 合作方ID
  partnerName: '张三',               // 合作方名称
  customerId: 'cust-001',           // 客户ID
  customerName: '张三',             // 客户名称
  supplier: '张三',                  // 供应商名称
  note: '',                         // 备注
  status: 'confirmed',              // 状态 (confirmed/settled)
  
  // 单据明细
  items: [
    {
      id: 'item-1234567890-0',      // 明细ID
      categoryId: 'cat-001',        // 品类ID
      categoryName: '布条类',       // 品类名称
      fabricId: 'fab-001',          // 布料ID
      fabricName: '布条类-90×115',  // 布料名称
      quantity: 100,                // 数量
      weightInput: '100',           // 重量输入文本
      totalWeight: 100,             // 总重量(斤)
      packCount: 0,                 // 包数
      packWeight: 0,                // 每包重量
      unit: '斤',                   // 单位
      unitPrice: 0.85,              // 单价(元/斤)
      amount: 85,                   // 金额(元)
      note: ''                      // 备注
    }
  ],
  
  // 汇总信息
  totalWeight: 100,                // 总重量(斤)
  totalAmount: 85,                 // 总金额(元)
  paidAmount: 0,                   // 已付金额
  receivedAmount: 0,               // 已收金额
  unsettledAmount: 85,             // 未结算金额
  
  // 过磅信息
  firstWeight: 0,                  // 皮重
  lastWeight: 0,                   // 毛重

```

#### 状态枚举
- **type**: `purchase`(进货)/`sale`(出货)
- **status**: `confirmed`(已确认)/`settled`(已结算)

#### 计算字段说明
- **unsettledAmount**：自动计算，当 <= 0 时状态变为 settled
- **totalWeight**：自动计算，取明细汇总或用户输入值
- **totalAmount**：自动计算，取明细汇总或用户输入值

## 数据流向

### 1. 客户管理
- **创建客户**：UI → CustomerStore.addCustomer() → localStorage.set(wsbs_customers)
- **查询客户**：CustomerStore.customers → 响应式列表
- **导出/导入**：支持 Excel 格式导入导出

### 2. 品类管理
- **创建品类**：UI → CategoryStore.addCategory() → localStorage.set(wsbs_categories)
- **查询品类**：CategoryStore.categories → 响应式列表

### 3. 布料管理
- **创建布料**：UI → FabricStore.addFabric() → localStorage.set(wsbs_fabrics)
- **查询布料**：FabricStore.fabrics → 响应式列表
- **价格管理**：支持批量更新采购和销售价格

### 4. 单据管理
- **创建单据**：UI → BillRecordStore.createRecord() → normalizeRecord() → localStorage.set(wsbs_bills)
- **查询单据**：BillRecordStore.records → 响应式列表
- **删除单据**：BillRecordStore.deleteRecord() → localStorage.remove()
- **修改单据**：BillRecordStore.updateRecord() → localStorage.set()

## 数据安全与持久化

### 1. 数据序列化
所有数据均通过 JSON.stringify 进行序列化存储，读取时通过 JSON.parse 解析。

### 2. 数据加密
系统目前未对 LocalStorage 数据进行加密。建议在生产环境中：
- 使用 HTTPS 协议传输数据
- 对敏感信息进行加密存储
- 实现定期清理机制

### 3. 错误恢复
- 数据加载失败时提供默认数据
- 定期备份机制（未实现）
- 数据验证和修复（在 normalizeRecord 函数中实现）

### 4. 数据过期策略
LocalStorage 数据永不过期。系统会在以下情况清理数据：
- 用户主动删除
- 应用清除缓存
- 浏览器清除缓存

## 性能优化

### 1. 数据缓存策略
- 统计数据缓存在 wsbs_stats_cache
- 最近使用记录缓存在 wsbs_recent_* 键中
- 常用查询结果通过 Pinia 的 computed 属性缓存

### 2. 数据精简
- 持久化时只保存必要字段
- 移除不必要的重复字段
- 定期清理过期和无效数据

### 3. 索引优化
- 使用 id 字段快速查找记录
- 按类型和日期排序提高查询效率
- 使用 computed 属性进行数据筛选

## 初始化数据
系统在首次启动时会自动初始化以下数据：
1. 用户信息（默认登录账号）
2. 4个示例客户
3. 2个品类
4. 32种布料
5. 示例单据（来自 billRecords.js）

## 数据迁移
目前系统不支持数据迁移。如需迁移数据：
1. 导出当前数据为 JSON 文件
2. 修改数据结构
3. 重新导入系统

## 总结
本系统采用简单但高效的数据存储方案，使用 LocalStorage 作为主要存储介质，结合 Pinia 状态管理实现数据的响应式更新。数据模型设计合理，涵盖了布碎资源结算业务的核心需求。

系统通过统一的 storage 工具类实现数据的序列化和反序列化，并提供了完整的数据验证和修复机制。所有核心业务功能都建立在这些数据模型之上，确保了数据的一致性和可靠性。
